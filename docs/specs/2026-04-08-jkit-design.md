# jkit — Design Spec

**Date:** 2026-04-08
**Status:** Draft

---

## Overview

A private Claude Code plugin for a Java / Spring Boot / Spring Cloud SaaS microservice
team, supporting both modern (3.1+) and legacy Spring Boot versions. It encodes the team's full development workflow — from spec change
to tested, committed implementation — as a set of skills, hooks, and CLI integrations.

The core loop:
```
edit docs/ spec files → commit → /spec-delta → clarify → plan → implement → test → commit
```

The `.spec-sync` marker file tracks which spec commit has been fully implemented,
so the delta is always precise regardless of how many commits have landed since.

Every microservice the team creates or migrates follows this layout:


### Configuration philosophy

- Single `application.yml` using `${ENV_VAR:default}` substitution
- Spring profile selection is implicit: load the correct `.env/<env>.env` file
- `source .env/test.env && mvn test` for integration tests
- No `application-{profile}.yml` files

### `.spec-sync`

- Plain text file containing the git SHA of the last spec commit that was
  fully implemented and committed
- Initialized to `HEAD` by `migrate-project`
- Read by `spec-delta` to compute the diff baseline
- Updated by the post-commit hook after each `feat(impl):` / `fix(impl):` / `chore(impl):` commit

### Commit conventions

| Prefix | Meaning |
|---|---|
| `docs(spec):` or `docs(<domain>):` | Spec change (processed by spec-delta) |
| `feat(impl):` | New feature implementation |
| `fix(impl):` | Bug fix implementation |
| `chore(impl):` | Non-feature implementation work |
| `chore(migrate):` | Project migration commit |

The `(impl):` scope is the trigger for the post-commit hook to update `.spec-sync`.
Spec commit format is a convention — `spec-delta` processes all commits that modify
`docs/` regardless of message format.

## Superpowers Compatibility

This plugin runs alongside the `superpowers` plugin. The following rules govern
how the two interact to avoid conflicts.

### Resolved conflicts

**1. `superpowers:brainstorming` must NOT be invoked from `migrate-project`**

The brainstorming skill has a hard-gate: it always ends by invoking
`superpowers:writing-plans`, and it runs a full design session (Q&A → design →
spec → plan). Invoking it from `migrate-project` just to resolve subdomain
boundary ambiguity would trigger this full flow unintentionally.

**Fix:** `migrate-project` handles subdomain ambiguity with its own targeted
questions (labeled options + recommendation). `superpowers:brainstorming` is
never invoked from this plugin.

**2. `superpowers:writing-plans` plan location must be overridden**

`writing-plans` defaults to saving plans at `docs/superpowers/plans/YYYY-MM-DD-*.md`.
Our plugin puts everything for a run in `docs/jkit/<run>/`.

**Fix:** `spec-delta` explicitly instructs `writing-plans` to save the plan to
`docs/jkit/<run>/plan.md`. This override is supported by `writing-plans`
("User preferences for plan location override this default").

**3. `publishing-service-contract` codeskel path**

The original `generate-microservice-skill` looks for `codeskel` on `PATH` only.
The plugin bundles `codeskel` in `bin/` — PATH may not include it on all machines.

**Fix:** `publishing-service-contract` resolves `codeskel` from `bin/` first,
then falls back to `PATH` (same resolution logic as the `comment` skill).

### Safe integrations

| Our skill | Superpowers skill used | How |
|---|---|---|
| `spec-delta` | `superpowers:writing-plans` | Invoked with plan location override |
| `java-tdd` | `superpowers:test-driven-development` | Extended after GREEN phase only |
| `migrate-project` | `superpowers:dispatching-parallel-agents` | Standard invocation |

### `java-coding-standards.md`

Java coding rules live in `<plugin-root>/docs/java-coding-standards.md`.
They are loaded explicitly at step 0 of code-writing skills only:
- `java-tdd` — step 0: read `java-coding-standards.md`
- `contract-testing` — step 0: read `java-coding-standards.md`

Non-code skills (`spec-delta`, `migrate-project`, `publishing-service-contract`,
`comment`) never load this file. The project `CLAUDE.md` (from `templates/CLAUDE.md`)
contains only workflow conventions — no coding rules.

## Skill Design Principles

**Questions to the human must include labeled options with a recommendation.**
Every time a skill asks the human to make a choice, it must:
- Offer 2-3 labeled options (A, B, C)
- Mark one as recommended with `(recommended)`
- Keep the default answer to one keystroke where possible

Example:
> "Should bulk invoice creation be transactional or best-effort?
> A) Transactional — all succeed or all fail (recommended)
> B) Best-effort — process valid items, skip invalid ones"

This applies to all skills in this plugin.

## Skills

### `spec-delta`

**Purpose:** Computes the requirements delta from spec files and drives implementation.

**Run directory:** Each invocation creates `docs/jkit/YYYY-MM-DD-<feature>/`
where `<feature>` is a short slug derived from the most significant change in the diff
(e.g., `billing-bulk-invoice`). All review artifacts for this run live there.

**Flow:**
1. Check if `docs/.spec-sync` exists
   - **Missing:** Initialize baseline interactively:
     a. Run `git log --oneline -- docs/` to find commits that touched spec files
     b. **No such commits:** Initialize silently to HEAD and continue:
        *"No spec commits found. Initialized .spec-sync to HEAD."*
     c. **Commits found:** Show the last 5, ask:
        *"No .spec-sync found. Which commit was the last one fully implemented?
         A) [sha] [date] [message]   ← most recent spec commit
         B) [sha] [date] [message]
         ...
         Z) HEAD — all current specs are already implemented
         C) Enter a specific SHA manually"*
        Initialize `.spec-sync` to the chosen SHA, then stop:
        *"Baseline set to [sha]. Run /spec-delta again to see what's pending."*
2. Read `docs/.spec-sync` → baseline SHA
3. `git diff <SHA> HEAD -- docs/` → list of changed spec files
4. If diff is empty → stop: *"No spec changes since last implementation."*
5. Check `docs/overview.md`:
   - **Missing:** Generate it before proceeding:
     Read all spec files in `docs/` → draft a ≤1 page overview describing what the
     service does, its domains, and primary responsibilities → ask targeted questions
     if anything is unclear → show draft to human for approval → write approved version
   - **Present:** Read it as background context
   - **New domain added in diff:** After change-summary approval, prompt:
     *"A new domain was added. Should docs/overview.md be updated?
      A) Yes — generate an updated draft (recommended)
      B) No — overview is still accurate"*
     If yes: draft update → show to human → write on approval
6. Identify which domains changed; order tasks within each domain by dependency:
   `domain-model` → `api-implement-logic` → `api-spec`
   Cross-domain ordering: if domain-A's model is referenced by domain-B's API,
   domain-A tasks must come first. Ask the human if cross-domain dependencies are unclear.
7. **Semantic schema change analysis:** Read the full diff of all three spec docs and
   reason about whether the changes imply database schema changes — new tables, new or
   renamed columns, new indexes, dropped columns, changed relationships. Do not rely on
   keyword scanning; use understanding of the domain model changes to determine this.
8. Ask targeted clarification questions one at a time — only for genuine ambiguities
   (with labeled options + recommendation on each question)
9. Write `docs/jkit/<run>/change-summary.md`:
   - Table of domains changed, what is added/modified/removed per domain
   - SQL migration flag: whether schema changes are implied (from step 7 analysis)
   - Cross-domain effects if any
10. Ask human to review and approve `change-summary.md` before proceeding
11. Invoke `superpowers:writing-plans` with: diff content + overview + clarification answers,
    instructing it to save the plan to `docs/jkit/<run>/plan.md`
12. After implementation is complete and committed, the post-commit hook updates
    `.spec-sync` automatically

**Invoked by:** `/spec-delta` command

### SQL migration

**Not a standalone skill** — SQL migration generation is a step triggered within
the `spec-delta` / `java-tdd` workflow when schema changes are detected.

**Trigger:** Flagged by the semantic schema change analysis in `spec-delta` step 7.
Claude reads the full diff of all three spec docs and reasons about whether the changes
imply database schema changes — no keyword matching, pure semantic understanding of what
the domain model changes mean for the underlying data structures.

**Flow (runs after `change-summary.md` is approved):**
1. Infer schema changes from diff across all three docs
2. Write `docs/jkit/<run>/migration-preview.md`:
   ```markdown
   ## Migration Preview: billing-bulk-invoice

   | Change | Type | Detail |
   |---|---|---|
   | `bulk_invoice` | CREATE TABLE | id, tenant_id, status, created_at |
   | `invoice.bulk_id` | ADD COLUMN | FK to bulk_invoice, nullable |
   | `idx_invoice_bulk` | CREATE INDEX | on invoice(bulk_id) |
   ```
3. Ask human to review and approve `migration-preview.md`
   - Options: **A) Approve as-is (recommended)** / B) Edit preview first / C) Skip migration
4. Generate SQL into `docs/jkit/<run>/migration/V<YYYYMMDD>_NNN__<feature>.sql`
5. As the final implementation step, move SQL file to `src/main/resources/db/migration/`
   and include it in the implementation commit

### `java-tdd`

**Purpose:** TDD workflow for Java, extended with JaCoCo coverage gap analysis.

**Wraps:** `superpowers:test-driven-development`

**Prerequisite:** `pom.xml` must include the JaCoCo Maven plugin configured to
generate XML reports. If missing, add the fragment from `templates/pom-fragments/jacoco.xml`.

**Extended flow (after GREEN phase):**
1. `mvn clean test jacoco:report`
   - **If command fails or `target/site/jacoco/jacoco.xml` is absent after build:**
     Stop and ask: *"JaCoCo report generation failed. Verify pom.xml includes
     the JaCoCo plugin (see templates/pom-fragments/jacoco.xml). Add it and re-run."*
2. `jacoco-filter target/site/jacoco/jacoco.xml --summary --min-score 1.0`
3. Feed JSON output to Claude → write tests for high-priority uncovered methods
4. Repeat steps 1-3 until no methods remain above score threshold
5. Final commit message must use prefix `feat(impl):`, `fix(impl):`, or `chore(impl):`

**CLI dependency:** `jacoco-filter` — shipped as a pre-built binary in `bin/`.
No installation required; the skill resolves the binary from the plugin's `bin/`
directory automatically.

### `contract-testing`

**Purpose:** API scenario tests derived from a specific domain's `api-spec.yaml`.

**Invocation:** Per-domain. The skill is invoked after implementing a domain's
endpoints. It targets one domain at a time.

**Spring Boot version detection:**
The skill reads `<parent><version>` from `pom.xml` and selects the testing strategy:

| Spring Boot version | Testing strategy |
|---|---|
| 3.1+ | `@SpringBootTest` + Testcontainers (`@ServiceConnection`) + RestAssured |
| < 3.1 | `docker-compose -f docker-compose.test.yml up` → RestAssured against running container |

**Testing stack — Spring Boot 3.1+:**
- **RestAssured** — HTTP assertions against `@SpringBootTest(webEnvironment = RANDOM_PORT)`
- **Testcontainers** — real infrastructure via `@ServiceConnection`
- **WireMock** — stub external microservice calls

**Testing stack — Spring Boot < 3.1:**
- **docker-compose.test.yml** — spins up service + all dependencies as containers
- **RestAssured** — HTTP assertions against the running service port
- **WireMock** — stub external microservice calls (run as sidecar in compose file)

**Prerequisite (3.1+):** `pom.xml` must include Testcontainers, RestAssured, and
WireMock test dependencies. If missing, add from `templates/pom-fragments/testcontainers.xml`.

**Test naming convention:** `*IntegrationTest.java` (e.g., `InvoiceIntegrationTest.java`)

**Human review step:** Before generating Java test code, the skill produces
`docs/jkit/<run>/contract-tests.md` — a human-readable table of all
planned test scenarios. The human reviews and edits this table, then the skill
generates Java code from the approved scenarios. This ensures coverage matches
intent and provides a permanent, readable record of what is tested.

**`contract-tests.md` format:**
```markdown
## Contract Tests: billing domain

| Endpoint | Scenario | Input | Expected |
|---|---|---|---|
| POST /invoices/bulk | happy path | valid list | 201 + list of invoice IDs |
| POST /invoices/bulk | empty list | [] | 400 validation error |
| POST /invoices/bulk | unauthenticated | no token | 401 |
| POST /invoices/bulk | missing required field | list without amount | 422 |
```

**Scope:** Only endpoints added or modified in the diff of `api-spec.yaml` for this
run. Unchanged endpoints already have tests and are not regenerated.

**Flow:**
1. Read the diff of `docs/domain-<n>/api-spec.yaml` from the run's baseline SHA
2. Extract only added or modified endpoints from the diff
3. Detect Spring Boot version from `pom.xml`
4. Ask: *"Which external services do these endpoints call?
   A) None (recommended if self-contained)
   B) [list detected Feign clients from codebase]"*
5. Write `docs/jkit/<run>/contract-tests.md` — scenarios for changed
   endpoints only
6. Ask human to review and approve the scenario table
7. Generate Java test code from the approved scenarios
8. Run tests:
   - 3.1+: `source .env/test.env && mvn test -Dtest=*IntegrationTest`
   - < 3.1: `docker-compose -f docker-compose.test.yml up -d && mvn test -Dtest=*IntegrationTest`

### `migrate-project`

**Purpose:** Bootstraps the standard structure on an existing Java microservice
that lacks docs, has sparse comments, and does not follow the team layout.

**Dependencies:**
- `codeskel domains` subcommand (see CLI section)
- `comment` skill (bundled in this plugin, uses `codeskel` from `bin/`)
- `superpowers:dispatching-parallel-agents`

**Parallelization scope:** Tasks (a)-(d) below run sequentially within each
subdomain. Different subdomains are processed in parallel — one agent per subdomain
via `superpowers:dispatching-parallel-agents`.

**Flow:**
1. `codeskel domains <project_root>` → detected subdomains JSON
   - **If no subdomains detected:** Ask: *"No subdomains detected automatically.
     How many logical domains does this service have? Name them and describe
     their primary responsibility."* Use the human's answer to define domains manually.
   - **If CLI fails:** Fall back to manual domain definition via the same question.
2. Present detected subdomains to human; ask for confirmation/corrections
   - If boundaries are ambiguous, ask targeted questions with labeled options
     (never invoke `superpowers:brainstorming` — see Superpowers Compatibility)
3. For each confirmed subdomain (parallel dispatch, sequential within each):
   a. Run `comment` skill on subdomain's classes
   b. Generate `api-spec.yaml` from `@RestController` classes in `api/`
   c. Generate `domain-model.md` from `@Entity` / `@Aggregate` classes
   d. Generate `api-implement-logic.md` from `@Service` classes
4. Copy template stubs: `example.env`, `docker-compose.yml`
   Generate `docs/overview.md`:
   - Analyze codeskel output + key classes (main class, controllers, domain entities)
   - Ask targeted questions if service purpose is unclear (e.g. "What is the primary
     business domain of this service? A) [inferred from packages] B) Other")
   - Draft ≤1 page overview → show to human for approval → write on approval
5. Add missing `pom.xml` plugin fragments from `templates/pom-fragments/` if absent
6. Extract all `${VAR}` references from `application.yml` → populate `example.env`
7. Initialize `docs/.spec-sync` to current `HEAD`
8. Final commit: `chore(migrate): bootstrap standard project structure`

**Invoked by:** `/migrate-project` command

## Hooks

### `post-commit-sync.sh`

Updates `.spec-sync` after implementation commits. Folds the update into the
implementation commit via `--amend` so no extra commit appears.

```bash
#!/usr/bin/env bash
set -euo pipefail

MSG=$(git log -1 --pretty=%s)

if echo "$MSG" | grep -qE '^(feat|fix|chore)\(impl\):'; then
  # Ensure docs/ directory exists
  mkdir -p docs

  # Write current HEAD SHA to .spec-sync
  git rev-parse HEAD > docs/.spec-sync || {
    echo "ERROR: Failed to write docs/.spec-sync" >&2
    exit 1
  }

  git add docs/.spec-sync

  # Only amend if .spec-sync actually changed
  if ! git diff --cached --quiet; then
    git commit --amend --no-edit || {
      echo "ERROR: Failed to amend commit with .spec-sync update." >&2
      echo "To recover: run 'git commit --amend --no-edit' manually," >&2
      echo "or 'git reset HEAD docs/.spec-sync' to unstage and skip." >&2
      exit 1
    }
  fi
fi
```

**Registered in** `hooks/hooks.json` as a `post-commit` hook.

## `codeskel domains` PRD

**Problem:** Migrating existing Java microservices to the standard doc structure
requires identifying subdomain boundaries. Reading all Java files with Claude is
token-expensive and imprecise. A deterministic scanner eliminates this cost.

**Approach:** Add a `domains` subcommand to the existing `codeskel` CLI. All
required data (packages, class-level annotations) is already collected by
`codeskel scan` via tree-sitter. No new tool or new parsing logic needed.

**Detection rules:**

| Signal | Role |
|---|---|
| Package name segments (`com.example.billing`) | Primary grouping key |
| `@RestController` base path prefix (`/billing/**`) | Primary confirmation |
| `@Entity` / `@Aggregate` package grouping | Secondary |
| `@Service` package grouping | Secondary |

**CLI interface:**
```
codeskel domains <project_root> [OPTIONS]

Options:
  --output <path>       Write JSON to file (default: stdout)
  --pretty              Pretty-print output
  --min-classes <n>     Minimum classes to qualify as a subdomain [default: 2]
  --src <path>          Override source root [default: src/main/java]
```

**Exit codes:** Follows existing `codeskel` conventions (0 = success, 1 = fatal,
2 = partial success with warnings).

**Non-goals:**
- Does not read method bodies
- Does not infer business logic or domain relationships
- Does not generate docs (that is Claude's responsibility)

**Implementation notes:**
- Reuse `.codeskel/cache.json` if present; run `codeskel scan` first if not
- Group by top-level package segment after the group ID prefix
  (e.g., `com.newland` prefix → `billing` from `com.newland.billing`)
- Annotation detection reuses existing tree-sitter parsing — no raw regex
- Rust, same codebase as `codeskel`

## Plugin Installation

**For team members:**
```bash
# 1. Clone the plugin repo
git clone <team-gitlab-url>/jkit ~/.claude/plugins/jkit

# 2. No CLI tool installation needed — binaries are in bin/ and used directly
chmod +x ~/.claude/plugins/jkit/bin/*

# 3. Remove personal skills now bundled in the plugin (if previously installed)
rm -rf ~/.claude/skills/comment/
rm -rf ~/.claude/skills/generate-microservice-skill/

# 4. Reload Claude Code plugins
# Run: /reload-plugins in Claude Code
```

**Updating:**
```bash
cd ~/.claude/plugins/jkit && git pull && chmod +x bin/*
```

**For plugin maintainers (building binaries):**
```bash
# Build and copy binaries after new releases of jacoco-filter or codeskel
# Linux / macOS
cargo build --release --manifest-path /path/to/jacoco-filter/Cargo.toml
cargo build --release --manifest-path /path/to/codeskel/Cargo.toml
cp target/release/jacoco-filter bin/jacoco-filter-$(uname -s | tr A-Z a-z)-$(uname -m)
cp target/release/codeskel      bin/codeskel-$(uname -s | tr A-Z a-z)-$(uname -m)

# Windows (cross-compile or build natively)
cargo build --release --target x86_64-pc-windows-gnu
cp target/x86_64-pc-windows-gnu/release/jacoco-filter.exe bin/jacoco-filter-windows-x86_64.exe
cp target/x86_64-pc-windows-gnu/release/codeskel.exe      bin/codeskel-windows-x86_64.exe
```

## Workflow: End-to-End Example
