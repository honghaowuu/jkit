**Date:** 2026-04-08
**Status:** Draft

## Overview

A private Claude Code plugin for a Java / Spring Boot / Spring Cloud SaaS microservice
team, supporting both modern (3.1+) and legacy Spring Boot versions. It encodes the team's full development workflow ‚Äî from spec change
to tested, committed implementation ‚Äî as a set of skills, hooks, and CLI integrations.

The core loop:
```
edit docs/ spec files ‚Üí commit ‚Üí /spec-delta ‚Üí clarify ‚Üí plan ‚Üí implement ‚Üí test ‚Üí commit
```

The `.spec-sync` marker file tracks which spec commit has been fully implemented,
so the delta is always precise regardless of how many commits have landed since.

### Configuration philosophy

- Single `application.yml` using `${ENV_VAR:default}` substitution
- Spring profile selection is implicit: load the correct `.env/<env>.env` file
- `.envrc` (committed) contains `dotenv ".env/${JKIT_ENV:-local}.env"` ‚Äî selects env file by `JKIT_ENV`, defaults to `local`
- `direnv allow` is run automatically by the jkit `SessionStart` hook ‚Äî no manual step required
- direnv auto-loads `local.env` when the developer enters the project directory; Claude Code inherits these vars as a child process ‚Äî no explicit loading needed for local commands
- To switch environment, re-evaluate `.envrc` via `direnv exec` with `JKIT_ENV` set:
  - Test: `JKIT_ENV=test direnv exec . <cmd>`
  - Dev: `JKIT_ENV=dev direnv exec . <cmd>`
- No `application-{profile}.yml` files

### `.spec-sync`

- Plain text file containing the git SHA of the last spec commit that was
  fully implemented and committed
- Initialized to `HEAD` by `migrate-project`
- Read by `spec-delta` to compute the diff baseline
- Updated by the post-commit hook after each `feat(impl):` / `fix(impl):` / `chore(impl):` commit

### Commit conventions

| Prefix | Meaning |
|---|---|
| `docs(spec):` or `docs(<domain>):` | Spec change (processed by spec-delta) |
| `feat(impl):` | New feature implementation |
| `fix(impl):` | Bug fix implementation |
| `chore(impl):` | Non-feature implementation work |
| `chore(migrate):` | Project migration commit |

The `(impl):` scope is the trigger for the post-commit hook to update `.spec-sync`.
Spec commit format is a convention ‚Äî `spec-delta` watches only `docs/domains/*/`
regardless of message format. Generated artifacts (`docs/jkit/`, `docs/skills/`) and
`docs/overview.md` are excluded.

## Superpowers Dependency

**Requires:** `superpowers` plugin (v5.0+). jkit does not function correctly without it.

**Why:** superpowers' `SessionStart` hook injects the skill-discovery behavior
("always check for skills before doing anything") that all jkit skills depend on.
jkit adds a token-free `SessionStart` hook for direnv setup only ‚Äî it outputs `{}`
with no context payload. Session context injection is fully delegated to superpowers.

### Resolved conflicts

**1. `superpowers:brainstorming` must NOT be invoked from `migrate-project`**

The brainstorming skill has a hard-gate: it always ends by invoking
`superpowers:writing-plans`, and it runs a full design session (Q&A ‚Üí design ‚Üí
spec ‚Üí plan). Invoking it from `migrate-project` just to resolve subdomain
boundary ambiguity would trigger this full flow unintentionally.

**Fix:** `migrate-project` handles subdomain ambiguity with its own targeted
questions (labeled options + recommendation). `superpowers:brainstorming` is
never invoked from this plugin.

**2. `superpowers:writing-plans` plan location must be overridden**

`writing-plans` defaults to saving plans at `docs/superpowers/plans/YYYY-MM-DD-*.md`.
Our plugin puts everything for a run in `docs/jkit/<run>/`.

**Fix:** `spec-delta` explicitly instructs `writing-plans` to save the plan to
`docs/jkit/<run>/plan.md`. This override is supported by `writing-plans`
("User preferences for plan location override this default").

**3. `publishing-service-contract` codeskel path**

The original `generate-microservice-skill` looks for `codeskel` on `PATH` only.
jkit bundles `codeskel` in `bin/` ‚Äî skills call it directly as `bin/codeskel`
with no resolution logic. Presence is validated once per session by the
`session-start` hook.

### Safe integrations

| Our skill | Superpowers skill used | How |
|---|---|---|
| `spec-delta` | `superpowers:writing-plans` | Plan location + header note overridden |
| `java-tdd` | `superpowers:test-driven-development` | Extended after GREEN phase only |
| `java-tdd` | `superpowers:subagent-driven-development` | Subagent-driven execution mode |
| `java-tdd` | `superpowers:executing-plans` | Inline execution mode |
| `java-tdd` | `superpowers:requesting-code-review` | After all tasks pass quality + coverage gates |
| `contract-testing` | `superpowers:debugging` | On implementation-bug test failures |
| `migrate-project` | `superpowers:dispatching-parallel-agents` | Standard invocation |

### `java-coding-standards.md`

Java coding rules live in `<plugin-root>/docs/java-coding-standards.md`.
They are loaded explicitly at step 0 of code-writing skills only:
- `java-tdd` ‚Äî step 0: read `java-coding-standards.md`
- `contract-testing` ‚Äî step 0: read `java-coding-standards.md`

Non-code skills (`spec-delta`, `migrate-project`, `publishing-service-contract`,
`comment`) never load this file. The project `CLAUDE.md` (from `templates/CLAUDE.md`)
contains only workflow conventions ‚Äî no coding rules.

## Skill Design Principles

**Questions to the human must include labeled options with a recommendation.**
Every time a skill asks the human to make a choice, it must:
- Offer 2-3 labeled options (A, B, C)
- Mark one as recommended with `(recommended)`
- Keep the default answer to one keystroke where possible

Example:
> "Should bulk invoice creation be transactional or best-effort?
> A) Transactional ‚Äî all succeed or all fail (recommended)
> B) Best-effort ‚Äî process valid items, skip invalid ones"

This applies to all skills in this plugin.

**Review artifacts are written to file ‚Äî path only.**
After writing any file the human must review, output only the file path. Do not show
the file content in the response. The human opens the file to review.

Example:
> "Written to `docs/jkit/2026-04-08-billing-bulk-invoice/change-summary.md`
> A) Looks good (recommended)
> B) Edit ‚Äî tell me what to change"

This applies to all review points: `change-summary.md`, `migration-preview.md`,
migration SQL, `contract-tests.md`, `plan.md`, and `generated-docs-review.md`.

## Skill Development

### Skill pattern

All jkit skills follow the superpowers skill pattern. When writing or modifying a skill:

- **Frontmatter:** YAML block with `name` and `description`. Description drives the Skill tool's trigger matching ‚Äî write it as triggering conditions, not a summary of what the skill does.
- **Process flow:** Every skill must have a `## Process Flow` section with a `dot` digraph showing the major steps and decision points. Decision nodes use `[shape=diamond]`, action nodes use `[shape=box]`, terminal states use `[shape=doublecircle]`. This gives Claude a navigable map of the skill before reading the detailed prose.
- **Checklist block:** Every skill must have a `## Checklist` section listing each major step as a `- [ ]` item. Checklist items are **task names**, not workflow descriptions ‚Äî short, action-oriented, suitable as a `TaskCreate` title (e.g., `"Detect Spring Boot version"`, not `"Detect Spring Boot version from pom.xml <parent><version> tag"`). Conditions and details belong in the process flow and prose, not in the task name. When Claude invokes the skill, it calls `TaskCreate` for each checklist item before starting work, then marks each `in_progress` / `completed` as it goes.
- **Labeled options:** Every question to the human must offer 2‚Äì3 labeled options (A, B, C) with one marked `(recommended)`.
- **HARD-GATE blocks:** Use `<HARD-GATE>` for mandatory human review points that must not be skipped.
- **Anti-rationalization tables:** For steps that are commonly skipped, add a rationalization table ("do NOT skip when: ...") modeled on the superpowers TDD skill's Red Flags pattern.
- **TaskCreate scope:** `TaskCreate` is for skill checklist items only ‚Äî one task per `## Checklist` line, created before work starts, managed by the superpowers injection pattern. Loop iterations within a step must NOT create additional tasks (this pollutes the task list with mixed granularities and confuses the checklist layer). Track loop progress with a file-based note (e.g., a `## Progress` section appended to the run's markdown artifact).

### Reference directory

`reference/jacoco-filter.md` and `reference/codeskel.md` are CLI reference docs maintained alongside the plugin source during development. When writing or modifying a skill that uses these tools, read the relevant reference file first to verify correct flag names, output formats, and exit codes. These files are **not shipped in the final plugin** ‚Äî they are author aids only and should be excluded from the plugin distribution.

### Skill origins and required refactoring

The following skills were adapted from existing personal skills. They must be **rewritten** to
the superpowers pattern ‚Äî not copy-pasted with minor edits. The personal skill is the reference
for workflow logic only; structure, frontmatter, and compliance gates must be built fresh.

| Skill | Source | Changes on merge |
|---|---|---|
| `comment` | `~/.claude/skills/comment/SKILL.md` | codeskel from `bin/` first; add `## Checklist`; add HARD-GATE before writing any file; add anti-rationalization table for `codeskel rescan` step |
| `publishing-service-contract` | `~/.claude/skills/generate-microservice-skill/SKILL.md` | codeskel from `bin/`; controller path convention to `src/main/java/com/newland/<service>/api/`; add `## Checklist`; add HARD-GATE before overwriting existing skill; add anti-rationalization table for Javadoc quality gate |

**What "rewritten to superpowers pattern" means for each skill:**
- `description` frontmatter written as **triggering conditions only** ‚Äî not a workflow summary
- `## Checklist` section with one `- [ ]` item per major step
- `<HARD-GATE>` blocks at mandatory human review points (e.g. "do not proceed until user approves")
- Anti-rationalization table for any step that is commonly skipped (e.g. "do NOT skip codeskel rescan when: ...")

## Skills

### `spec-delta`

**Purpose:** Computes the requirements delta from spec files and drives implementation.

**Run directory:** Each invocation creates `docs/jkit/YYYY-MM-DD-<feature>/`
where `<feature>` is a short slug derived from the most significant change in the diff
(e.g., `billing-bulk-invoice`). All review artifacts for this run live there.

**Resume:** If a run directory already exists (artifacts from a previous interrupted
run), spec-delta detects it and asks:
> "Found an existing run `2026-04-08-billing-bulk-invoice` with incomplete artifacts.
> A) Resume from where it stopped (recommended)
> B) Start a fresh run"
On resume, read existing artifacts to determine which steps are already done and
continue from the first incomplete step. Claude Code's `/resume` command restores
conversation context; file-based artifacts restore workflow position.

**Flow:**
1. Sync with remote:
   a. `git fetch`
   b. `git rev-list HEAD..@{u} --count` ‚Äî check if remote is ahead
   c. **Remote not ahead:** continue
   d. **Remote ahead, working tree clean:** `git pull --ff-only` and continue
   e. **Remote ahead, working tree dirty:** stop and ask:
      *"Remote has new commits but you have local changes. How do you want to proceed?
       A) Stash local changes, pull, then unstash (recommended)
       B) Continue without pulling ‚Äî work from current local state
       C) Abort ‚Äî I'll resolve this manually"*
2. Check if `docs/.spec-sync` exists
   - **Missing:** Initialize baseline interactively:
     a. Run `git log --oneline -- docs/domains/*/` to find commits that touched spec files
     b. **No such commits:** Initialize silently to HEAD and continue:
        *"No spec commits found. Initialized .spec-sync to HEAD."*
     c. **Commits found:** Show the last 5, ask:
        *"No .spec-sync found. Which commit was the last one fully implemented?
         A) [sha] [date] [message]   ‚Üê most recent spec commit
         B) [sha] [date] [message]
         ...
         Z) HEAD ‚Äî all current specs are already implemented
         C) Enter a specific SHA manually"*
        Initialize `.spec-sync` to the chosen SHA, then stop:
        *"Baseline set to [sha]. Run /spec-delta again to see what's pending."*
3. Read `docs/.spec-sync` ‚Üí baseline SHA
4. `git diff <SHA> HEAD -- docs/domains/*/` ‚Üí list of changed spec files
5. If diff is empty ‚Üí stop: *"No spec changes since last implementation."*
6. Check `docs/overview.md`:
   - **Missing:** Generate it before proceeding:
     Read all spec files in `docs/domains/*/` ‚Üí draft a ‚â§1 page overview describing what the
     service does, its domains, and primary responsibilities ‚Üí ask targeted questions
     if anything is unclear ‚Üí write draft to `docs/overview.md` ‚Üí tell human path ‚Üí
     ask: `"A) Looks good (recommended) B) Edit ‚Äî tell me what to change"` ‚Üí repeat on B
   - **Present:** Read it as background context
   - **New domain added in diff:** After change-summary approval, prompt:
     *"A new domain was added. Should docs/overview.md be updated?
      A) Yes ‚Äî generate an updated draft (recommended)
      B) No ‚Äî overview is still accurate"*
     If yes: write updated draft ‚Üí tell human path ‚Üí get approval as above
7. Identify which domains changed; order tasks within each domain by dependency:
   `domain-model` ‚Üí `api-implement-logic` ‚Üí `api-spec`
   Cross-domain ordering: if domain-A's model is referenced by domain-B's API,
   domain-A tasks must come first. Ask the human if cross-domain dependencies are unclear.
8. **Semantic schema change analysis:** Read the full diff of all three spec docs and
   reason about whether the changes imply database schema changes ‚Äî new tables, new or
   renamed columns, new indexes, dropped columns, changed relationships. Do not rely on
   keyword scanning; use understanding of the domain model changes to determine this.
9. Ask targeted clarification questions one at a time ‚Äî only for genuine ambiguities
   (with labeled options + recommendation on each question)
10. Create run directory `docs/jkit/YYYY-MM-DD-<feature>/` where `<feature>` is a short slug
    derived from the most significant change in the diff (e.g., `billing-bulk-invoice`)
11. Write `docs/jkit/<run>/change-summary.md`:
    - Table of domains changed, what is added/modified/removed per domain
    - SQL migration flag: whether schema changes are implied (from step 8 analysis)
    - Cross-domain effects if any
12. Tell human: `"Written to docs/jkit/<run>/change-summary.md ‚Äî A) Looks good (recommended) B) Edit ‚Äî tell me what to change"` ‚Äî repeat on B
13. **If schema changes were flagged in step 8:** run the SQL migration sub-flow
    (see `### SQL migration` section) ‚Äî introspect DB, write `migration-preview.md`,
    get human approval, generate SQL, get human approval, then continue
14. Invoke `superpowers:writing-plans` with: diff content + overview + clarification answers,
    and two overrides:
    - Save plan to `docs/jkit/<run>/plan.md` (not the superpowers default location)
    - Replace the plan header agentic-worker note with:
      `> **For agentic workers:** Use `/java-tdd` to implement this plan (TDD workflow with JaCoCo coverage analysis).`
15. Tell human: `"Plan written to docs/jkit/<run>/plan.md ‚Äî A) Looks good (recommended) B) Edit ‚Äî tell me what to change"`
    - On B: apply requested changes to `plan.md` ‚Üí repeat
    - On A: invoke `java-tdd` directly ‚Äî `java-tdd` will ask execution mode (Subagent-Driven or Inline?)
16. After implementation is complete and committed, the post-commit hook updates
    `.spec-sync` automatically

### SQL migration

**Not a standalone skill** ‚Äî SQL migration generation is a step triggered within
the `spec-delta` workflow when schema changes are detected.

**Trigger:** Flagged by the semantic schema change analysis in `spec-delta` step 8.
Claude reads the full diff of all three spec docs and reasons about whether the changes
imply database schema changes ‚Äî no keyword matching, pure semantic understanding of what
the domain model changes mean for the underlying data structures.

**Flow (runs after `change-summary.md` is approved):**
1. Infer schema changes from diff across all three docs
2. **Live schema introspection** ‚Äî detect database type from `pom.xml` JDBC driver
   (`postgresql`, `mysql`, etc.), then run read-only `information_schema` queries
   using credentials already in the environment (loaded by `direnv` from `.env/local.env`):
   ```bash
   # PostgreSQL example ‚Äî $DATABASE_URL already in environment via direnv auto-load
   psql "$DATABASE_URL" -t -c "
     SELECT column_name, data_type, is_nullable, column_default
     FROM information_schema.columns
     WHERE table_name = 'invoice'
     ORDER BY ordinal_position;"
   ```
   No credential handling in the skill ‚Äî local env vars are already inherited from the parent shell.
   This reveals columns already applied by other services, so the generated migration
   is a true delta against the live schema, not against the spec alone.

   **Fallback ‚Äî env vars not in environment:** direnv may not be hooked into the shell,
   or Claude Code was started outside the project directory. Read `.env/local.env` directly.

   **Fallback ‚Äî database unreachable:** warn:
   *"Could not connect to the local database. Migration will be generated from the spec
   alone ‚Äî review carefully before applying, as the table may already have some of
   these columns."* Then continue with step 3 using spec-inferred schema only.

3. Write `docs/jkit/<run>/migration-preview.md` using the introspected delta:
   ```markdown
   ## Migration Preview: billing-bulk-invoice

   | Change | Type | Detail |
   |---|---|---|
   | `bulk_invoice` | CREATE TABLE | id, tenant_id, status, created_at |
   | `invoice.bulk_id` | ADD COLUMN | FK to bulk_invoice(id), nullable |
   | `idx_invoice_bulk` | CREATE INDEX | on invoice(bulk_id) |
   ```
   Columns that already exist in the live database are omitted from the preview.

4. Tell human: `"Written to docs/jkit/<run>/migration-preview.md ‚Äî A) Approve as-is (recommended) B) Edit preview first C) Skip migration"`
5. Generate SQL into `docs/jkit/<run>/migration/V<YYYYMMDD>_NNN__<feature>.sql`
6. Tell human: `"Migration SQL written to docs/jkit/<run>/migration/<file>.sql ‚Äî A) Looks good ‚Äî move to src/main/resources/db/migration/ (recommended) B) Edit the SQL first C) Abort"`
7. On approval, move SQL file to `src/main/resources/db/migration/`
   and include it in the implementation commit

### `java-tdd`

**Purpose:** Plan-driven TDD workflow for Java, extended with JaCoCo coverage gap
analysis. Reads the latest spec-delta plan and implements each task via RED/GREEN/REFACTOR.
Can also be used standalone (without a plan) for ad-hoc TDD work.

**Wraps:** `superpowers:test-driven-development`

**Prerequisite:** `pom.xml` must include the JaCoCo Maven plugin configured to
generate XML reports. If missing, add the fragment from `templates/pom-fragments/jacoco.xml`.

**Quality gate prerequisite:** At least one quality plugin (Checkstyle, PMD, or SpotBugs)
is recommended. If none are present, the quality gate step will offer to add all three
from `templates/pom-fragments/quality.xml`:

```xml
<!-- Checkstyle -->
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-checkstyle-plugin</artifactId>
  <version>3.3.1</version>
  <configuration>
    <configLocation>google_checks.xml</configLocation>
    <failsOnError>true</failsOnError>
    <consoleOutput>true</consoleOutput>
  </configuration>
</plugin>

<!-- PMD -->
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-pmd-plugin</artifactId>
  <version>3.21.0</version>
  <configuration>
    <failOnViolation>true</failOnViolation>
    <printFailingErrors>true</printFailingErrors>
  </configuration>
</plugin>

<!-- SpotBugs -->
<plugin>
  <groupId>com.github.spotbugs</groupId>
  <artifactId>spotbugs-maven-plugin</artifactId>
  <version>4.8.3.1</version>
  <configuration>
    <effort>Max</effort>
    <threshold>Medium</threshold>
    <failOnError>true</failOnError>
  </configuration>
</plugin>
```

**Resume:** If invoked after an interruption, determine progress from durable state:
check `git log --oneline` for `feat(impl):`/`fix(impl):` commits since the run baseline,
then cross-reference against the plan tasks. Continue from the first task with no
corresponding commit ‚Äî no prompt needed, just announce which task is being resumed.

**Plan detection:** On invocation, check for the most recent spec-delta run directory
under `docs/jkit/`. If `plan.md` exists there and `.spec-sync` has not yet been updated
to HEAD, offer:
> "Found plan `docs/jkit/<run>/plan.md`. Implement from this plan?
> A) Yes ‚Äî implement per plan (recommended)
> B) No ‚Äî ad-hoc TDD (I'll describe what to build)"

**Execution handoff (plan-driven only):** After the human confirms the plan, scan
the plan tasks to assess coupling: are tasks self-contained (each adds a distinct
feature with its own types), or tightly coupled (tasks share interfaces, build on
each other's scaffolding)? Then offer:

> "How should I implement the plan?
> 1. **Subagent-Driven** ‚Äî dispatch a fresh subagent per task via
>    `superpowers:subagent-driven-development`; each subagent runs the full
>    RED/GREEN/REFACTOR cycle + JaCoCo coverage gap analysis before completing.
>    Best for loosely coupled tasks where each task adds a self-contained feature.
> 2. **Inline** (recommended for this plan) ‚Äî implement tasks sequentially in this
>    session via `superpowers:executing-plans`, with TDD checkpoints and JaCoCo gap
>    analysis after each task.
>    Best for tightly coupled tasks that share interfaces or build on each other's
>    scaffolding ‚Äî a fresh subagent would need to re-read significant context each time.
>
> *(Recommended: [1 or 2 based on coupling assessment])*"

When plan-driven, read the full plan before starting. Implement tasks in the order
specified by the plan.

**Extended flow (after GREEN phase for each task):**
1. **Quality gate** ‚Äî scan `pom.xml` for Checkstyle, PMD, SpotBugs plugins:
   - If none found: offer to add from `templates/pom-fragments/quality.xml`
     - Human declines: skip quality gate and continue to step 2
     - Human accepts: add fragment, then run
   - Run detected tools: `mvn checkstyle:check pmd:check spotbugs:check` (only the ones present)
   - Fix failures inline; if new plugins were added, note them in the final commit message
2. `mvn clean test jacoco:report`
   - **If command fails or `target/site/jacoco/jacoco.xml` is absent after build:**
     Stop and ask: *"JaCoCo report generation failed. Verify pom.xml includes
     the JaCoCo plugin (see templates/pom-fragments/jacoco.xml). Add it and re-run."*
3. `jacoco-filter target/site/jacoco/jacoco.xml --summary --min-score 1.0`
4. If gaps remain above threshold: for each method in priority order, invoke
   `superpowers:test-driven-development` targeting that specific uncovered path.
   Each gap is a mini TDD cycle: write a test that exercises the missing branch (RED) ‚Üí
   confirm it fails for the right reason ‚Üí make it pass (GREEN).
5. Repeat steps 2‚Äì4 until `jacoco-filter` reports no methods above the score threshold.
6. Invoke `superpowers:requesting-code-review`
7. Final commit message must use prefix `feat(impl):`, `fix(impl):`, or `chore(impl):`

**CLI dependency:** `jacoco-filter` ‚Äî invoked via `bin/jacoco-filter` wrapper.
No installation or OS detection required; the wrapper selects the correct binary automatically.

### `contract-testing`

**Purpose:** API scenario tests derived from a specific domain's `api-spec.yaml`.

**Invocation:** Per-domain. The skill is invoked after implementing a domain's
endpoints. It targets one domain at a time.

**Spring Boot version detection:**
The skill reads `<parent><version>` from `pom.xml` and selects the testing strategy:

| Spring Boot version | Testing strategy |
|---|---|
| 3.1+ | `@SpringBootTest` + Testcontainers (`@ServiceConnection`) + RestAssured |
| < 3.1 | compose file ‚Üí RestAssured against running container |

**Container runtime resolution (< 3.1 path only):**
Resolve in order:
1. `docker` ‚Äî use `docker compose` (plugin syntax) or `docker-compose` (standalone)
2. `podman` ‚Äî use `podman compose`
3. Neither found ‚Üí stop and ask:
   *"No container runtime found. Please install Docker or Podman, then re-run."*

**Testing stack ‚Äî Spring Boot 3.1+:**
- **RestAssured** ‚Äî HTTP assertions against `@SpringBootTest(webEnvironment = RANDOM_PORT)`
- **Testcontainers** ‚Äî real infrastructure via `@ServiceConnection`
- **WireMock** ‚Äî stub external microservice calls

**Testing stack ‚Äî Spring Boot < 3.1:**
- **docker-compose.test.yml** ‚Äî spins up service + all dependencies as containers (via resolved runtime)
- **RestAssured** ‚Äî HTTP assertions against the running service port
- **WireMock** ‚Äî stub external microservice calls (run as sidecar in compose file)

**Prerequisite (3.1+):** `pom.xml` must include Testcontainers, RestAssured, and
WireMock test dependencies. If missing, add from `templates/pom-fragments/testcontainers.xml`.

**Test naming convention:** `*IntegrationTest.java` (e.g., `InvoiceIntegrationTest.java`)

**Human review step:** Before generating Java test code, the skill produces
`docs/jkit/<run>/contract-tests.md` ‚Äî a human-readable table of all
planned test scenarios. The human reviews and edits this table, then the skill
generates Java code from the approved scenarios. This ensures coverage matches
intent and provides a permanent, readable record of what is tested.

**`contract-tests.md` format:**
```markdown
## Contract Tests: billing domain

| Endpoint | Scenario | Input | Expected |
|---|---|---|---|
| POST /invoices/bulk | happy path | valid list | 201 + list of invoice IDs |
| POST /invoices/bulk | empty list | [] | 400 validation error |
| POST /invoices/bulk | unauthenticated | no token | 401 |
| POST /invoices/bulk | missing required field | list without amount | 422 |
```

**Scope:** Only endpoints added or modified in the diff of `api-spec.yaml` for this
run. Unchanged endpoints already have tests and are not regenerated.

**Flow:**
1. Read the diff of `docs/domains/<name>/api-spec.yaml` from the run's baseline SHA
2. Extract only added or modified endpoints from the diff
3. Detect Spring Boot version from `pom.xml`
4. Ask: *"Which external services do these endpoints call?
   A) None (recommended if self-contained)
   B) [list detected Feign clients from codebase]"*
5. Write `docs/jkit/<run>/contract-tests.md` ‚Äî scenarios for changed
   endpoints only
6. Tell human: `"Written to docs/jkit/<run>/contract-tests.md ‚Äî A) Looks good (recommended) B) Edit ‚Äî tell me what to change"` ‚Äî repeat on B
7. Generate Java test code from the approved scenarios:
   - **‚â§ ~20 total scenarios across all controllers:** generate the full test class in
     one pass.
   - **> ~20 or multiple controllers with many scenarios:** split by controller ‚Äî
     one `*IntegrationTest.java` per controller, processed sequentially. Append a
     `## Generation Progress` checklist to `contract-tests.md` before starting:
     ```markdown
     ## Generation Progress
     - [ ] InvoiceController ‚Üí InvoiceIntegrationTest.java
     - [ ] PaymentController ‚Üí PaymentIntegrationTest.java
     ```
     After generating and testing each controller's class, mark its line `[x]`.
     This survives context compression and allows resuming mid-loop without TaskCreate.
8. Run tests:
   - **One-shot (all classes at once):**
     - 3.1+: `JKIT_ENV=test direnv exec . mvn test -Dtest=*IntegrationTest`
     - < 3.1: `<runtime> compose -f docker-compose.test.yml up -d && JKIT_ENV=test direnv exec . mvn test -Dtest=*IntegrationTest`
   - **Split (per-controller loop):** run after generating each class:
     - 3.1+: `JKIT_ENV=test direnv exec . mvn test -Dtest=<ControllerName>IntegrationTest`
     - < 3.1: containers already up from first iteration ‚Äî run only: `JKIT_ENV=test direnv exec . mvn test -Dtest=<ControllerName>IntegrationTest`
   - Container runtime runs directly ‚Äî it does not need test env vars. Only Maven needs the test environment.
9. **If tests fail ‚Äî classify before acting:**
   - **A) Generated code error** (compilation failure, assertion doesn't match the
     approved scenario table): fix the generated test inline ‚Äî do not change the
     approved scenario table. Re-run tests.
   - **B) Implementation bug** (test is correct per spec, production endpoint
     misbehaves): invoke `superpowers:debugging` targeting the failing endpoint.
   - **After one self-fix pass:** if tests still fail for any reason, always escalate
     to `superpowers:debugging` regardless of failure type.

### Checklist

- [ ] Load java-coding-standards
- [ ] Detect Spring Boot version
- [ ] Resolve container runtime
- [ ] Verify test dependencies
- [ ] Extract changed endpoints from api-spec diff
- [ ] Ask about external dependencies
- [ ] Write contract-tests.md
- [ ] Get scenario approval
- [ ] Generate test code
- [ ] Run tests
- [ ] Fix failures or invoke superpowers:debugging
- [ ] Confirm all tests passing

(`java-tdd` via its final commit step, or the human directly).

## Hooks

### `session-start` (SessionStart hook)

Ensures `direnv` is installed and the current project's `.envrc` is allowed.
Token-free ‚Äî no context injected. jkit skills are triggered explicitly by the developer
(via slash commands or direct requests), so no session-level workflow injection is needed.

```bash
#!/usr/bin/env bash
# hooks/session-start
set -euo pipefail

PLUGIN_ROOT="$(cd "$(dirname "$0")/.." && pwd)"

# 1. Install direnv if missing
if ! command -v direnv &>/dev/null; then
  echo "jkit: direnv not found ‚Äî installing..." >&2
  curl -sfL https://direnv.net/install.sh | bash 2>/dev/null \
    || { echo "jkit: direnv install failed. Please install manually: https://direnv.net" >&2; }
fi

# 2. Allow the current project's .envrc (idempotent ‚Äî persists by .envrc hash)
if command -v direnv &>/dev/null && [ -f ".envrc" ]; then
  direnv allow . 2>/dev/null || true
fi

# 3. Validate bundled CLI tools ‚Äî warn once if missing or not executable
for tool in codeskel jacoco-filter; do
  if [ ! -x "${PLUGIN_ROOT}/bin/${tool}" ]; then
    echo "jkit: bin/${tool} missing or not executable. Run: chmod +x ~/.claude/plugins/jkit/bin/*" >&2
  fi
done

printf '{}\n'
exit 0
```

**`direnv allow` is idempotent:** keyed by `.envrc` hash ‚Äî silent no-op when already
allowed, auto-re-approves when `.envrc` is updated by the team.

**How env vars reach Claude Code:** direnv auto-loads `.envrc` when the developer enters
the project directory. Claude Code, started from that shell, inherits the loaded env vars
as a child process ‚Äî so local env vars are available to all bash commands without any
explicit loading. For non-local environments (test, dev), `JKIT_ENV=<env> direnv exec .`
re-evaluates `.envrc` with the target env, overriding the inherited local vars.

**Registered in** `hooks/hooks.json` as a `SessionStart` hook.

### `post-commit-sync.sh`

Updates `.spec-sync` after implementation commits. Folds the update into the
implementation commit via `--amend` so no extra commit appears.

```bash
#!/usr/bin/env bash
set -euo pipefail

MSG=$(git log -1 --pretty=%s)

if echo "$MSG" | grep -qE '^(feat|fix|chore)\(impl\):'; then
  # Ensure docs/ directory exists
  mkdir -p docs

  # Write current HEAD SHA to .spec-sync
  git rev-parse HEAD > docs/.spec-sync || {
    echo "ERROR: Failed to write docs/.spec-sync" >&2
    exit 1
  }

  git add docs/.spec-sync

  # Only amend if .spec-sync actually changed
  if ! git diff --cached --quiet; then
    git commit --amend --no-edit || {
      echo "ERROR: Failed to amend commit with .spec-sync update." >&2
      echo "To recover: run 'git commit --amend --no-edit' manually," >&2
      echo "or 'git reset HEAD docs/.spec-sync' to unstage and skip." >&2
      exit 1
    }
  fi
fi
```

**Re-entrancy guard:** On some Git versions, amending inside `post-commit` triggers
a second `post-commit` run. The script is safe because the second run finds `.spec-sync`
already at HEAD (`git rev-parse HEAD` equals current content), so `git diff --cached --quiet`
returns true and no second amend occurs. Validate this on the team's Git version before
deploying; add a `JKIT_HOOK_RUNNING=1` env var guard if re-entrancy is observed.

**Registered in** `hooks/hooks.json` as a `post-commit` hook alongside the `SessionStart` entry:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [{ "type": "command", "command": "${CLAUDE_PLUGIN_ROOT}/hooks/session-start", "async": false }]
      }
    ]
  },
  "post-commit": "hooks/post-commit-sync.sh"
}
```

## `codeskel domains` PRD

**Problem:** Migrating existing Java microservices to the standard doc structure
requires identifying subdomain boundaries. Reading all Java files with Claude is
token-expensive and imprecise. A deterministic scanner eliminates this cost.

**Approach:** Add a `domains` subcommand to the existing `codeskel` CLI. All
required data (packages, class-level annotations) is already collected by
`codeskel scan` via tree-sitter. No new tool or new parsing logic needed.

**Detection rules:**

| Signal | Role |
|---|---|
| Package name segments (`com.example.billing`) | Primary grouping key |
| `@RestController` base path prefix (`/billing/**`) | Primary confirmation |
| `@Entity` / `@Aggregate` package grouping | Secondary |
| `@Service` package grouping | Secondary |

**CLI interface:**
```
codeskel domains <project_root> [OPTIONS]

Options:
  --output <path>       Write JSON to file (default: stdout)
  --pretty              Pretty-print output
  --min-classes <n>     Minimum classes to qualify as a subdomain [default: 2]
  --src <path>          Override source root [default: src/main/java]
```

**Exit codes:** Follows existing `codeskel` conventions (0 = success, 1 = fatal,
2 = partial success with warnings).

**Non-goals:**
- Does not read method bodies
- Does not infer business logic or domain relationships
- Does not generate docs (that is Claude's responsibility)

**Implementation notes:**
- Reuse `.codeskel/cache.json` if present; run `codeskel scan` first if not
- Group by top-level package segment after the group ID prefix
  (e.g., `com.newland` prefix ‚Üí `billing` from `com.newland.billing`)
- Annotation detection reuses existing tree-sitter parsing ‚Äî no raw regex
- Rust, same codebase as `codeskel`

## Workflow: End-to-End Example

```
RUN DIR: docs/jkit/2026-04-08-billing-bulk-invoice/

1. Developer edits docs/billing/domain-model.md (new BulkInvoice entity)
             and docs/billing/api-spec.yaml (new POST /invoices/bulk endpoint)
2. git commit -m "docs(billing): add bulk invoice creation"

3. /spec-delta
   ‚Üí computes diff since .spec-sync
   ‚Üí reads docs/overview.md for context
   ‚Üí asks: "Should bulk invoice creation be transactional or best-effort?
            A) Transactional ‚Äî all succeed or all fail (recommended)
            B) Best-effort ‚Äî process valid items, skip invalid ones"
   ‚Üí writes change-summary.md ‚Üí tells human: "Written to docs/jkit/.../change-summary.md ‚Äî A) Looks good B) Edit"
   ‚Üí human approves
   ‚Üí detects domain-model.md changed ‚Üí writes migration-preview.md ‚Üí human approves
   ‚Üí generates migration/V20260408_001__add_bulk_invoice.sql ‚Üí human approves
   ‚Üí invokes writing-plans ‚Üí writes plan.md ‚Üí tells human: "Written to docs/jkit/.../plan.md ‚Äî A) Looks good B) Edit"
   ‚Üí human approves plan ‚Üí invokes java-tdd directly (java-tdd asks execution mode)

4. Implementation (java-tdd per task):
   ‚Üí write failing test ‚Üí RED ‚Üí implement ‚Üí GREEN
   ‚Üí quality gate: pom.xml has Checkstyle ‚Üí mvn checkstyle:check ‚Üí PASS
   ‚Üí mvn clean test jacoco:report ‚Üí jacoco-filter ‚Üí no high-priority gaps
   ‚Üí superpowers:requesting-code-review ‚Üí review passes
   ‚Üí final task: mv migration SQL to src/main/resources/db/migration/

5. contract-testing (billing domain):
   ‚Üí reads api-spec.yaml
   ‚Üí writes contract-tests.md (4 scenarios for POST /invoices/bulk)
   ‚Üí human reviews and approves contract-tests.md
   ‚Üí generates InvoiceBulkIntegrationTest.java from approved table
   ‚Üí JKIT_ENV=test direnv exec . mvn test -Dtest=*IntegrationTest ‚Üí PASS

6. git commit -m "feat(impl): add bulk invoice creation endpoint"
   ‚Üí post-commit hook: updates docs/.spec-sync, amends commit

7. /publish-contract
   ‚Üí updates docs/skills/billing-service/SKILL.md with new endpoint
```

## Usage Scenarios

### A ‚Äî Implement a spec change (primary loop)

1. Developer edits `docs/domains/billing/domain-model.md` and `api-spec.yaml`
2. `git commit -m "docs(spec): add bulk invoice feature"`
3. `/spec-delta` ‚Üí syncs remote ‚Üí computes diff ‚Üí asks clarifications ‚Üí writes
   `change-summary.md` ‚Üí human approves ‚Üí if schema changes: writes `migration-preview.md`
   ‚Üí human approves ‚Üí generates SQL ‚Üí human approves ‚Üí writes `plan.md` ‚Üí plan review loop
   (review ‚Üí edit ‚Üí review ‚Üí ...) ‚Üí human approves ‚Üí invokes `java-tdd` directly
4. `java-tdd` ‚Üí detects plan ‚Üí implements tasks via TDD ‚Üí runs quality gate ‚Üí
   runs jacoco-filter ‚Üí fills coverage gaps ‚Üí invokes `superpowers:requesting-code-review`
   ‚Üí final `feat(impl):` commit ‚Üí post-commit hook updates `.spec-sync`
5. `/contract-testing` ‚Üí generates `contract-tests.md` ‚Üí human approves scenarios ‚Üí
   generates `*IntegrationTest.java` ‚Üí tests pass

**Potential issue:** writing-plans may prompt for implementation method. Fixed by
spec-delta telling writing-plans not to ask (implementation is always via java-tdd).

### B ‚Äî Bootstrap a legacy service

1. Developer has existing Java service without standard structure
2. `/migrate-project` ‚Üí detects subdomains ‚Üí comments classes ‚Üí generates
   `docs/domains/*/` spec files ‚Üí copies templates ‚Üí writes `generated-docs-review.md`
   ‚Üí human approves ‚Üí final `chore(migrate):` commit
3. `/publish-contract` ‚Üí generates `docs/skills/<service>/SKILL.md`
4. Developer edits generated spec files to add missing business context, then
   `git commit -m "docs(spec): refine initial spec"` ‚Üí run Scenario A from here

**Potential issue:** migrate-project generates api-spec from existing code ‚Äî it
reflects current behaviour, not intent. Docs review gate (step 2) is the only guard.

### C ‚Äî Another service wants to call this one

1. Team B developer clones or reads this service's repo
2. Reads `docs/skills/<service>/SKILL.md` ‚Äî gets capabilities, `use_when`, `nacos_name`,
   SDK dependency, and per-module API path prefixes
3. Greps openapi.yaml for the relevant path prefix to get request/response schemas
4. No direct code reading required

**Potential issue:** SKILL.md goes stale if endpoints change. `/publish-contract`
must be run after each spec change that affects the API.

### D ‚Äî Shared table schema change from another service

1. Team B adds a column to a shared table via their own migration (already applied to DB)
2. Team A's developer runs `/spec-delta` for a feature that also touches that table
3. spec-delta introspects live DB via `information_schema` ‚Üí sees Team B's column already
   exists ‚Üí migration-preview.md shows only the true delta ‚Üí no duplicate column migration

**Potential issue:** If DB is unreachable (no local DB running), spec-delta falls back
to spec-only migration ‚Äî may include columns that already exist. The SQL review gate
catches this before the file is moved to `src/main/resources/db/migration/`.

### E ‚Äî Resume after interruption

1. Developer runs `/spec-delta`, approves `change-summary.md`, then gets interrupted
   before `plan.md` is written
2. Developer re-runs `/spec-delta` ‚Üí detects existing run directory ‚Üí
   asks "Resume run `2026-04-08-billing-bulk-invoice`?" ‚Üí resumes from the
   `writing-plans` invocation step, skipping already-completed steps
3. Alternatively: use Claude Code's `/resume` to restore conversation context, then
   re-run the skill ‚Äî the run directory state prevents re-doing completed work

**Potential issue:** If `.spec-sync` was not yet updated (interrupted before commit),
re-running spec-delta after the resume commit would show an empty diff. Resume detection
handles this: it checks for an in-progress run directory before computing the diff.
