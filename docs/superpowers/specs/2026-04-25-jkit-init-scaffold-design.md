# jkit init: project scaffold flow

**Status:** Draft
**Date:** 2026-04-25

## Background

`/migrate-project` is jkit's bootstrap skill. Today it does only two mechanical things — `jkit changes bootstrap` (creates `docs/changes/{pending,done}/`) and `jkit standards init` (creates `docs/project-info.yaml`) — then prints a blurb telling the human to write their first change file.

The conventions documented elsewhere in the plugin (`hooks/jkit-context.md`, `docs/standards/environment.md`, `skills/spec-delta/SKILL.md`) require several other project-owned files to exist before any skill can run cleanly:

- `.envrc` (direnv loader; the session-start hook calls `direnv allow .` on it)
- `.env/local.env`, `.env/test.env` (env files; standards/environment.md requires them)
- `docker-compose.yml` (local-dev stack — distinct from the test stack `jkit scenarios prereqs` regenerates)
- `docs/overview.md` (≤1 page service description; cited by `spec-delta` Step 15 but never auto-created)
- `.gitignore` entries for `.env/`, `target/`, `.jkit/done/`

The plugin already ships scaffolding sources under `templates/` (`envrc`, `example.env`, `docker-compose.yml`) but no code copies them into a project — they are orphaned. As a result every onboarded project hand-creates these files inconsistently, drifts from the documented conventions, and trips `/scenario-tdd` and `/java-verify` later when one is missing.

This spec defines a **`jkit init` umbrella subcommand** that wraps the existing two bootstrap calls and adds a new **`jkit init scaffold`** step that copies the orphaned templates into the project. The skill `/migrate-project` collapses to a single binary call plus orientation prose.

## Goals

1. Add `jkit init` as the canonical one-shot bootstrap entry point. Idempotent — re-running on a partially bootstrapped project completes only what's missing.
2. Add `jkit init scaffold` that copies template files into the project, skipping any that already exist.
3. Keep `jkit changes bootstrap` and `jkit standards init` as standalone subcommands (back-compat for any caller that targets them).
4. Update `/migrate-project` to call `jkit init` and surface its JSON report.
5. Remove the orphan/dead-template smell flagged in the plugin audit by giving every shipped template a consumer.

## Non-Goals

- Generating a `pom.xml` skeleton. jkit assumes a Java project already exists (the session-start hook nudges to `/migrate-project` only when `pom.xml`/`build.gradle*` is present).
- Generating `application.yml` or `logback-spring.xml`. Spring Boot archetypes own those; the standards files prescribe their content, not their creation.
- Per-project overrides for the embedded templates (`pom-fragments/*.xml`, `docker-compose.test.yml`, contract `.tera` files). Those stay binary-bound — the audit confirmed embedding is the right call for templates the binary mutates into user files. A future `project-info.yaml > pom-fragments-overrides:` knob can land later if a real user demands it.
- Scaffolding the first change file. `/write-change` already owns that interaction.
- Authoring CI workflow files. Out of scope for v1; revisit once the GitHub Actions story is decided.

## File Layout

All scaffold templates are **owned by the binary**, embedded via `include_str!` at compile time, and live in the jkit-cli source tree:

```
jkit-cli/crates/jkit/templates/
├── scaffold/
│   ├── envrc                       → emitted as .envrc
│   ├── example.env                 → emitted as .env/local.env and .env/test.env
│   ├── docker-compose.yml          → emitted as docker-compose.yml (local-dev stack)
│   └── overview.md                 → emitted as docs/overview.md (skeleton)
├── docker-compose.test.yml         (regenerated by jkit scenarios prereqs)
├── pom-fragments/*.xml             (spliced into pom.xml by jkit pom prereqs)
└── contract/*.tera                 (rendered by jkit contract stage)
```

`scaffold/overview.md` ships with placeholder sections (purpose, scope, public surface, owned data, dependencies, runbook) and `<!-- TODO: ... -->` markers. Short on purpose — the spec for overview.md says "≤1 page".

The plugin repo's `templates/` directory does **not** carry duplicate copies. After this spec lands plus the parallel dead-template cleanup (separate commit, removes `templates/pom-fragments/*.xml`, `templates/docker-compose.test.yml`, `templates/docker-compose.yml`, `templates/envrc`, `templates/example.env`, and `skills/publish-contract/templates/*.tera`, all of which the audit confirmed are unreferenced duplicates of binary-embedded content), the plugin repo's top-level `templates/` directory is removed entirely. The architectural rule that survives:

- **Files Claude reads at runtime** (skill scaffolding) live as files under `skills/<name>/templates/` — e.g., `skills/scenario-tdd/templates/integration-test-sb31.java`. The SKILL.md cites them by path; Claude reads them as test-class scaffolding.
- **Files the binary mutates, applies, or emits** are embedded in the binary via `include_str!`. The binary is the only source of truth.

No third category, no duplicates anywhere.

## CLI Surface

### `jkit init`

```
$ jkit init
{
  "ok": true,
  "steps": {
    "changes_bootstrap": { "created": ["docs/changes/pending/.gitkeep", "docs/changes/done/.gitkeep"], "existing": [] },
    "standards_init":    { "created": ["docs/project-info.yaml"], "existing": [] },
    "scaffold":          {
      "created":  [".envrc", ".env/local.env", ".env/test.env", "docker-compose.yml", "docs/overview.md"],
      "existing": [],
      "gitignore_added": [".env/", "target/", ".jkit/done/"]
    }
  },
  "next_steps": [
    "Review docs/project-info.yaml and adjust enabled flags",
    "Fill TODOs in docs/overview.md",
    "Run /write-change to author your first change file"
  ]
}
```

Idempotent. Re-running on a fully bootstrapped project returns `ok: true` with empty `created` arrays everywhere; nothing is overwritten.

Flags:

| Flag | Default | Meaning |
|---|---|---|
| `--pretty` | off | pretty-print JSON |
| `--apply` | on | write changes (default — opposite of pom prereqs idiom because init is meant to be the one-shot path; explicit for symmetry) |
| `--dry-run` | off | mutually exclusive with `--apply`; reports what *would* happen |

### `jkit init scaffold`

The file-copy step in isolation, for callers (and humans) who already ran `changes bootstrap` and `standards init` separately. Same JSON shape as the `scaffold` sub-object above.

```
$ jkit init scaffold --dry-run
{
  "ok": true,
  "would_create":  [".envrc", ".env/local.env", ".env/test.env", "docker-compose.yml", "docs/overview.md"],
  "would_skip":    [],
  "gitignore_would_add": [".env/", "target/", ".jkit/done/"]
}
```

### Existing subcommands

`jkit changes bootstrap` and `jkit standards init` stay unchanged. `jkit init` calls them internally; nothing forces a caller to migrate.

## Scaffold Behavior

For each (template, target) pair:

| Target | Source | Action if target exists | Action if target missing |
|---|---|---|---|
| `.envrc` | `templates/envrc` | record in `existing[]`, skip | copy verbatim |
| `.env/local.env` | `templates/example.env` | record in `existing[]`, skip | copy with header rewritten to `# JKIT_ENV=local — fill in real values, do NOT commit` |
| `.env/test.env` | `templates/example.env` | record in `existing[]`, skip | copy with header rewritten to `# JKIT_ENV=test — fixtures for integration tests` |
| `docker-compose.yml` | `templates/docker-compose.yml` | record in `existing[]`, skip | copy verbatim |
| `docs/overview.md` | `templates/overview.md` | record in `existing[]`, skip | copy verbatim |

`.gitignore` is special-cased: idempotent line-merge. The binary reads existing `.gitignore` (creating it if absent), checks for each managed entry (`.env/`, `target/`, `.jkit/done/`), and appends only the missing ones under a fenced jkit block:

```
# --- jkit-managed (do not edit between markers) ---
.env/
target/
.jkit/done/
# --- end jkit-managed ---
```

Re-running on an already-fenced `.gitignore` is a no-op. Entries the user added outside the fence are preserved.

The `templates/example.env` rewrite (target-specific header) is the only template not copied byte-for-byte. All others are pure copies.

## Skill Update: `/migrate-project`

Replaces the current four-step Detailed Flow with three steps. Step numbering renamed for clarity (no semantic change beyond the new file-scaffold step folded into Step 3):

```
### Step 1 — Ensure git repo
git rev-parse --git-dir 2>/dev/null || git init -q

### Step 2 — Detect existing state
ls -d docs/changes/ pom.xml 2>/dev/null    # warn if no Java build file

### Step 3 — Bootstrap
jkit init                                  # single call replaces three

### Step 4 — Stage and orient
git add docs/ .envrc .env/ docker-compose.yml .gitignore
```

Orientation prose (the "what to do next" message) reads from the `next_steps[]` array in the JSON, so the skill prose and the binary stay in sync without duplication. The skill prints each entry verbatim plus the existing `/write-change` hand-off question.

The Checklist section drops two bullets (`jkit changes bootstrap`, ensure `docs/project-info.yaml`) and gains one (`jkit init`).

## Failure Modes

| Failure | Behavior | Exit code |
|---|---|---|
| `templates/<name>` missing from plugin (corrupt install) | Surface the missing path, abort, partial work preserved | 1 |
| Target file exists but is unreadable / unwriteable | Surface OS error, mark step as `partial`, report what completed | 2 |
| `.gitignore` contains the jkit fence but with wrong entries (user-edited) | Repair: rewrite the fenced block to the canonical set; entries outside fence untouched | 0 |
| `--dry-run --apply` simultaneously | Argument parse error before any work | 1 |
| Already in `docs/changes/done/` (project deeply bootstrapped) | All steps return `existing[]`, exit 0; nothing overwritten | 0 |

The exit-code conventions match the rest of the binary (0 = success, 1 = fatal, 2 = partial-success-with-warnings) per the README CLI reference.

## Implementation Outline

The plan that follows from this spec will sequence the work roughly as:

1. Add `jkit-cli/crates/jkit/templates/scaffold/{envrc,example.env,docker-compose.yml,overview.md}` to the binary's source tree. Author `overview.md` from scratch with the placeholder skeleton; copy the other three from the plugin repo as the seed before deletion.
2. Add `jkit init scaffold` to the Rust CLI: take template-target pairs as a static list, embed the four templates via `include_str!`. The example-env header rewrite (the only non-byte-for-byte copy) is a small string substitution applied at emit time.
3. Add `jkit init` as a thin orchestrator that calls `changes::bootstrap`, `standards::init`, then `init::scaffold` and merges JSON outputs.
4. Add `.gitignore` line-merge logic with the fence marker. Reuse a single helper for fence detection / rewrite.
5. Add a smoke test: empty fixture dir → `jkit init` → assert the expected files exist with expected content; re-run → assert all `created[]` arrays are empty.
6. Update `/migrate-project/SKILL.md` to call `jkit init` and consume `next_steps[]`. Update Checklist + Process Flow DOT.
7. Update `README.md` `jkit` subcommands table to add `jkit init` and `jkit init scaffold` rows; drop the `templates/` line from the Repo layout block once the directory is gone.
8. Delete the plugin-repo `templates/` directory in the same commit that ships the new binary. The directory contains only embedded-into-binary duplicates by then; deleting it eliminates the drift surface for good. (`skills/<name>/templates/` directories survive — those are runtime-read by Claude, not embedded in the binary.)

Detailed sequencing belongs in the implementation plan (writing-plans skill).

## Risks and Trade-offs

- **Plugin-repo `templates/` is removed entirely** (Implementation Outline step 8). Humans browsing the plugin repo can no longer see scaffold output by `cat`-ing a file; they get the same answer from `jkit init scaffold --dry-run` or by reading the jkit-cli source. Trade accepted because the alternative (keep duplicate copies + a discipline note) recreates the exact drift problem this audit just fixed for pom-fragments and contract templates. The architectural rule — *binary owns what it emits; skill `templates/` dirs own what Claude reads at runtime* — is simple enough to hold without policing.
- **`docker-compose.yml` content depends on `project-info.yaml`** (Postgres vs MySQL, Redis on/off). v1 ships a single static template (Postgres + Redis, both commented "remove if unused"). v2 can templatize off `project-info.yaml` once `jkit init` has a forward-and-backward order: `standards init` produces `project-info.yaml` *before* `scaffold` runs, so the template can read it. v1 keeps `scaffold` ignorant of `project-info.yaml` to keep ordering loose — listed as future work.
- **Idempotent re-run guarantees no overwrite, which means a user who deleted `.envrc` to regenerate must explicitly delete the file first.** No `--force` flag in v1; add only if requested. Today the worst case is "the file you wanted regenerated is now gone — copy from `templates/envrc` by hand."
- **`.gitignore` fence marker is a contract.** Users who manually edit between the markers will see their edits clobbered on next `jkit init`. The fence comment makes this explicit; the spec accepts that risk because the alternative (smart merging) is brittle. Mitigation: the fence content is small and stable (3 lines), so the loss surface is bounded.
- **Coupling onboarding success to `jkit init` JSON shape.** `/migrate-project` parses `next_steps[]` and prints it verbatim. If the binary's JSON shape changes, the skill breaks silently (wrong prose). Mitigation: the smoke test asserts the JSON shape; skill prose is one Read of the JSON, no parsing logic that can drift.
